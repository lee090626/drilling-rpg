## ⚠️ 2. 절대 금기 사항 (Strict Don'ts)
1.  **직접적인 DOM 접근 금지:** 게임 엔진은 Web Worker 환경에서 실행됩니다. UI 레이어를 제외한 곳에서 `window`나 `document` 객체를 절대 참조하지 마세요.
2.  **임시 이미지(Placeholder) 외부 링크 금지:** 이미지가 필요할 경우 `generate_image` 도구를 사용하며, `placehold.it` 등의 외부 URL은 절대 사용하지 마세요.
3.  **FSD (Feature-Sliced Design) 아키텍처 위반 금지:** 레이어 간 단방향 의존성 규칙(Upper → Lower)을 엄격히 준수하세요.
4.  **임의적인 코드 삭제 금지:** 사용하지 않는 로직이라도 즉시 삭제하지 말고, 주석 처리 후 인간 개발자에게 반드시 확인을 요청하세요.
5.  **절대적인 주석 사용:** 항상 작업을 할때에는 JS-DOC으로 상세한 주석을 작성합니다.
6.  **UI 스타일 제약:** 현대적이고 정갈한 디자인을 위해 `italic` 및 `uppercase` Tailwind 클래스 사용을 금지합니다. (사용자 강력 요청사항)

## 🏛️ 3. 핵심 아키텍처 (Architecture)
* **UI 레이어:** Next.js 15 + Tailwind CSS 4 (전역 상태 공유: Zustand)
* **로직 레이어:** Web Workers 활용 (ECS 패턴: System-Component 분리 설계)
* **데이터 동기화:** `Transferable Objects`를 활용한 Zero-copy 통신 구현

## 🐙 4. GitHub 협업 규칙 (Version Control)
* **브랜치 전략:** `{type}/{issue-number}-{description}` 형식을 엄격하게 따릅니다.
    * *예시:* `feat/12-shop-upgrade-logic`, `fix/45-worker-memory-leak`
* **커밋 메시지:** Conventional Commits 규약을 따르며, 상세 설명은 **한글**로 명확하게 작성합니다.
* **이슈 및 PR 템플릿 엄수:** 
    * 프로젝트에 정의된 템플릿 구조(`## 개요`, `## 주요 변경 사항` 등)를 절대로 생략하지 말고 충실히 작성하세요. 
    * `Closes #번호` 규칙을 반드시 준수하여 이슈를 PR에 연동하세요.
* **⚠️ CLI 사용 시 주의 (중요):** `gh pr create --body "..."` 등의 명령어를 터미널에 입력할 때, 따옴표 안에 문자열 `\n`을 날것으로 쓰면 GitHub에서 줄바꿈이 깨집니다. 반드시 실제 개행이 들어간 멀티라인 커맨드를 사용하거나 heredoc 구문을 활용하여 마크다운 포맷팅이 유지되도록 하세요.

## 🔄 5. 작업 프로세스 (Development Workflow)
> **중요:** 작업의 규모와 중요도에 따라 아래 두 가지 트랙 중 하나를 선택하여 진행합니다. 모든 경우에 **절대 경로** 사용은 필수입니다.

### 🛤️ 트랙 1: 표준 트랙 (Standard Track) - "중요 작업"
*신규 기능 개발, 엔진 로직 수정, 대규모 리팩토링 등 기록이 중요한 경우*
1.  **이슈 할당 및 확인**: `gh issue create` 또는 `docs/issues/` 문서를 통해 범위를 확정하고 이슈 번호를 할당받습니다.
    - *Tip:* `issue_no=$(gh issue create ... --json number --jq .number)` 명령어로 번호를 즉시 추출하여 브랜치 생성에 활용할 수 있습니다.
2.  **브랜치 생성 및 환경 설정**: `feat/{issue-number}-{description}` 형식으로 브랜치를 생성하고, 모든 작업은 **절대 경로**를 기준으로 수행합니다.
3.  **코드 구현 및 테스트**: FSD 아키텍처와 금기 사항을 준수하며 코드를 작성하고, 로컬 환경에서 검증합니다.
4.  **커밋 및 서버 동기화 (Push)**: Conventional Commits 규약에 맞춰 커밋 후, 반드시 `git push origin {branch-name}`을 실행합니다.
5.  **PR 제출 및 머지**: `gh pr create`를 통해 제출하고, 사용자 승인 후 메인 브랜치에 통합합니다.
    - *연결:* PR 본문에 `Closes #번호`를 포함하여 이슈와 자동으로 연결합니다.

### 🚀 트랙 2: 패스트 트랙 (Fast Track) - "경미한 작업"
*오타 수정, UI 상세 조정, 사소한 설정값 변경, 단순 버그 수정 등*
1.  **즉시 구현**: 별도의 이슈나 브랜치 생성 없이 즉시 코드 수정 및 검증을 수행합니다.
2.  **직접 커밋 및 푸시**: 현재 브랜치(또는 `main`)에 즉시 커밋하고 푸시합니다.
    *   *메시지 예시:* `fix: UI 간격 조정`, `docs: Agents.md 워크플로우 최적화`
3.  **사후 보고**: 작업 완료 후 사용자에게 작업 내역을 간략히 보고합니다.

### 🛠️ 공통 필수 사항
*   **절대 경로 사용**: 모든 명령 및 파일 접근 시 `/Users/ihyeongseog/drilling-game/` 기준 절대 경로 사용을 생활화합니다.
*   **환경 변수 관리**: `gh`, `node`, `npm` 사용 시 반드시 절대 경로(`/usr/local/bin/...` 등)를 확인하거나 `PATH`에 등록 후 실행합니다.
*   **오류 수정 보고**: 빌드 오류나 런타임 버그 수정 시, 사용자에게 **오류의 원인**과 **조치 내용**을 명확하게 설명합니다.
*   **빌드 검증**: 작업 완료 전 반드시 `npx tsc --noEmit`을 실행하여 Type Error를 0으로 유지합니다. (`npm run build`는 무겁기 때문에 가급적 지양합니다.)
*   **Agents.md 유지**: 작업 종료 시 `Status`를 비우거나 '완료'로 갱신하고 `Changelog`에 기록을 남깁니다.
